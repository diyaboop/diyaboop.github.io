Section 1

Overview

FraudWatch is an internal ML monitoring tool that trains a fraud detection classifier on financial transaction data, audits it for demographic bias, and tracks performance drift over time. It is designed for risk teams at financial services companies and responsible AI practitioners who need automated, auditable evidence that their fraud models are performing equitably across customer segments.

Section 2

Problem Statement

2.1 Background

Fraud detection models make high-stakes decisions affecting real customers. A false positive — wrongly flagging a legitimate transaction — blocks the customer's purchase, damages trust, and creates customer service overhead. When false positives are systematically concentrated on specific customer segments (by card type, product category, or geography), the model is not just inaccurate — it is discriminatory. This creates regulatory exposure under ECOA, EU AI Act Article 10, and CFPB model risk guidance.

2.2 Current State

  • Most fraud teams monitor aggregate precision and recall but do not segment performance by customer group
  • Bias audits, when they happen, are manual, infrequent, and not part of standard model deployment
  • There is no automated alerting when model performance degrades or when segment-level disparity exceeds acceptable thresholds
  • Compliance documentation for model fairness is assembled manually before audits rather than generated continuously

2.3 Opportunity

Automate the entire bias audit and monitoring workflow so risk teams get continuous, segment-level visibility into model fairness — not just aggregate accuracy. Make compliance documentation a byproduct of normal operations, not a pre-audit scramble.

Section 4

Users & User Stories

4.1 User Personas

Risk Analyst
Reviews model behavior post-deployment. Needs segment-level audit reports to identify where the model is misbehaving and prioritize remediation.
Compliance Officer
Responsible for regulatory filings and internal audits. Needs audit trails, statistical evidence, and regulatory mapping to demonstrate due diligence.
ML Engineer
Owns the model and retraining pipeline. Needs drift alerts and performance baselines to know when to retrain.
Product Manager
Responsible for the fraud detection product. Needs to understand precision/recall tradeoffs and customer impact to set thresholds and communicate with leadership.

4.2 User Stories

Epic 1
Bias Audit
  • As a Risk Analyst, I want to see false positive rates broken down by card type and product category so I can identify which customer segments are being disproportionately blocked and escalate for remediation.
  • As a Risk Analyst, I want to run a statistical significance test on segment disparities so I know whether the difference is real or random noise before escalating.
  • As a Compliance Officer, I want to export a structured bias audit report so I can include it in model risk documentation without manual compilation.
Epic 2
Performance Monitoring
  • As an ML Engineer, I want to receive an alert when weekly recall drops below 0.85 so I know immediately when the model needs retraining without manual monitoring.
  • As an ML Engineer, I want to see performance metrics broken down by week so I can identify when drift started and correlate it with data or behavioral changes.
  • As a Product Manager, I want to see a single dashboard showing recall, precision, and FPR trends so I can communicate model health to leadership in a weekly review.
Epic 3
Compliance Reporting
  • As a Compliance Officer, I want to see bias findings mapped to specific regulatory requirements so I can demonstrate during an audit that we are actively monitoring for discrimination.
  • As a Risk Analyst, I want to query the audit database with SQL so I can run custom analyses on specific time periods or segments without engineering support.

Section 6

Non-Functional Requirements

  • Performance — Full pipeline (train + audit + monitor) completes in under 15 minutes on standard hardware
  • Reproducibility — All results reproducible with fixed random_state=42 — same input always produces same output
  • Auditability — All audit results stored with timestamps in SQLite — full history retained indefinitely
  • Portability — Entire pipeline containerized with Docker — runs identically on any machine
  • Configurability — Alert thresholds (recall floor, FPR ceiling, disparity multiplier) configurable via CLI args

7.2 Data Flow

Raw CSVs (train_transaction.csv + train_identity.csv) → merge on TransactionID → threshold filter (drop >50% missing) → feature selection (30 features) → categorical encoding → missing value fill (-999) → train/test split (80/20, stratified) → Random Forest training → predictions → audit DataFrame → segment groupby → FPR calculation → z-test → SQLite storage → visualization → report generation.

Section 8

Acceptance Criteria

IDGiven / When / Then
AC-01Given a trained model, when audit.py runs, then FPR is calculated for every unique value of card4, card6, ProductCD, and addr2
AC-02Given two segments with FPR disparity, when z-test runs, then p-value and significance flag are returned for each pair
AC-03Given any segment with FPR > 2× overall average, when audit completes, then that segment is flagged in bias_summary.csv
AC-04Given 12 weeks of transaction data, when monitor.py runs, then precision, recall, and FPR are calculated and stored for each week
AC-05Given recall drops below 0.85 in any week, when monitor.py runs, then a drift alert is printed and flagged in the database
AC-06Given the full pipeline runs, when complete, then disparity_chart.png and drift_chart.png are saved to outputs/
AC-07Given the Docker container builds, when run with correct data volume mount, then full pipeline executes without errors
Section 9

Risks & Open Questions

RiskLevelMitigation
Data availability — IEEE-CIS dataset is 650MB, requires Kaggle accessMEDIUMProvide sample dataset for testing; document Kaggle download steps
Proxy discrimination — card4/card6/ProductCD are proxies, not protected classesHIGHDocument clearly; real protected attributes require additional data collection in production
Threshold sensitivity — 0.3 fraud probability threshold is manually chosen; different thresholds change bias findings significantlyMEDIUMMake threshold configurable via CLI; document sensitivity analysis
Model retraining — current model trained once; production retraining requires re-running auditLOWDocument retraining workflow; design audit to run automatically post-retrain

Open Questions

  • Should the disparity threshold (2×) be configurable or fixed? Who owns that decision in production?
  • What is the remediation workflow when bias is detected — who gets notified, what actions are taken?
  • Should the API require authentication? What data governance applies to audit results?
  • In production, how frequently should the audit run — on every retrain, weekly, or monthly?
Section 10

Scope (v1)

In Scope
  • Batch fraud classification on IEEE-CIS data
  • Segment-level FPR audit with z-test significance
  • 12-week drift monitoring with configurable alerts
  • SQLite audit trail with full history
  • CSV and PNG compliance exports
  • FastAPI layer for query access
  • Docker containerization
Out of Scope (v1)
  • Real-time transaction scoring
  • Multi-model comparison
  • Automated retraining on drift detection
  • Protected class analysis (race, gender, age)
  • Production-scale consumer traffic deployment
