Section 1

Overview

This document defines the product requirements for a three-module responsible AI audit pipeline applied to a convolutional neural network (CNN) trained on the CIFAR-10 image classification benchmark. The pipeline addresses a gap in standard ML evaluation: aggregate accuracy metrics do not reveal which classes are systematically underperformed, whether the model survives real-world perturbations, or whether its confidence scores are reliable signals.

The audit produces structured, evidence-based findings with statistical significance testing across three dimensions: class-level fairness (bias audit), input perturbation stability (robustness audit), and confidence reliability (calibration audit). All findings are visualized in a live public dashboard.

Section 2

Problem Statement

Standard image classifier evaluation reports a single accuracy number. This creates three blind spots:

  • Class bias: Aggregate accuracy hides per-class disparity. A model scoring 81% may perform at 91% on one class and 58% on another — a 33pp equity gap invisible in top-line reporting.
  • Robustness gaps: Models trained on curated datasets degrade under real-world perturbations (rotation, noise, lighting variation) without any warning in standard metrics.
  • Calibration failures: Models that express high confidence on wrong predictions mislead downstream systems that use confidence scores to gate automated action or human oversight.
The core problem: Deploying a model without auditing these three dimensions means deploying unknown risk. This pipeline makes that risk visible, measurable, and documentable before deployment decisions are made.

Epic 1
Bias Audit — Class-Level Fairness
  • Given a trained CNN and CIFAR-10 test set, When the bias audit runs, Then per-class accuracy is computed and displayed for all 10 classes with best/worst highlighted.
  • Given per-class accuracy data, When the gap between best and worst class exceeds 10pp, Then a Z-test is computed and the result flagged with p-value.
  • Given audit results, When a user views the dashboard, Then a bar chart ranks all classes by accuracy with a visual threshold line at the mean.
Epic 2
Robustness Audit — Perturbation Stability
  • Given a clean test set, When rotation (15°), gaussian noise (σ=0.1), and brightness shift (+0.2) are applied, Then accuracy is measured under each perturbation independently.
  • Given perturbation results, When any perturbation causes >5pp accuracy drop, Then it is flagged as a robustness failure in the dashboard.
  • Given robustness findings, When a user views the dashboard, Then a comparison chart shows baseline vs post-perturbation accuracy side-by-side for each test.
Epic 3
Calibration Audit — Confidence Reliability
  • Given model predictions with softmax confidence scores, When the calibration audit runs, Then the count and percentage of predictions with >90% confidence is computed and segmented by correct vs incorrect.
  • Given calibration results, When wrong predictions with >90% confidence exceed 5% of all high-confidence predictions, Then a calibration failure warning is displayed.
  • Given calibration findings, When a user views the dashboard, Then a confidence distribution histogram shows the shape of confidence scores across correct and incorrect predictions.

Section 6

Non-Functional Requirements

  • Reproducibility: Full pipeline (train → audit → dashboard) reproducible with a single command from a clean environment
  • No backend dependency: Dashboard must be fully static — hosted on GitHub Pages with no server-side runtime
  • Statistical validity: All significance claims must use appropriate two-sample Z-tests with correct degrees of freedom
  • Modularity: Each audit module (bias, robustness, calibration) must be independently runnable without executing the full pipeline
  • Accessibility: Dashboard must be publicly accessible without authentication or local setup

Data Flow

CIFAR-10 (via PyTorch torchvision) → CNN training (train.py) → model.pth saved → three audit modules run independently against saved weights → findings written to audit_results.json → Chart.js dashboard reads JSON at page load → GitHub Pages serves static HTML/JS dashboard at public URL.

Key Technical Choices

  • PyTorch for model training — native CIFAR-10 DataLoader, GPU-compatible, standard for CV research
  • SciPy for Z-test significance testing — scipy.stats.proportions_ztest with Bonferroni correction for multiple comparisons
  • Chart.js for dashboard — no backend, renders in browser from JSON, embeds cleanly in GitHub Pages
  • Matplotlib / Seaborn for static audit visualizations generated during pipeline run

Open Questions

  • Should robustness be tested per-class (e.g., does rotation hurt cat classification more than automobile) or only aggregate?
  • Is temperature scaling (post-hoc calibration fix) in scope for v1 or deferred to v2?
  • Should the dashboard support uploading a new model.pth to re-run audits dynamically, or remain static?

Section 11

Responsible AI Context

This project is itself a responsible AI artifact — not just an application of responsible AI practices. It demonstrates that model evaluation is incomplete without fairness, robustness, and calibration analysis, and provides a replicable template for applying this standard to any image classifier.

PrincipleHow This Project Addresses It
Fairness Per-class bias audit with statistical significance testing surfaces and quantifies unequal treatment across categories
Robustness Perturbation testing documents model behavior under conditions that deviate from controlled training distribution
Transparency All findings publicly accessible via live dashboard; methodology documented in code and PRD
Accountability Statistical evidence (Z-stat, p-value) enables audit findings to be challenged, replicated, and included in model risk documentation
Human Oversight Calibration audit specifically addresses whether model confidence can be trusted to gate automated vs. human-reviewed decisions
